** What is a Programming Log?
:PROPERTIES:
:CUSTOM_ID: what-is-programming-log
:END:
#+LaTeX: \index{programming log!definition}

A programming log is a thinking document for planning, tracking, and reflecting on the development of a software project.
It serves as an external repository for metadata, decisions, and progress notes that would otherwise clutter source code comments or get lost in scattered notes.
This document supports the metacognition required for effective software development.

The programming log differs from inline code comments and README files because it captures the /why/ behind decisions, not just the /what/.
It provides a safe place for exploratory thinking, abandoned approaches, and lessons learned that do not belong in version-controlled source code.

** Purpose of This Document
:PROPERTIES:
:CUSTOM_ID: purpose
:END:
#+LaTeX: \index{purpose}

This programming log supports the following workflows:

- Project ideation and specification (initial planning sessions)
- Architecture and design decision recording
- Daily progress logging to capture decisions, obstacles, and achievements
- Periodic project assessment against timeline and milestones
- Project completion, documentation, and archiving

** Why Keep a Programming Log?
:PROPERTIES:
:CUSTOM_ID: why-keep-log
:END:
#+LaTeX: \index{benefits}

- Eases resumption of work after interruptions (vacation, competing projects, emergencies)
- Supports concurrent work on multiple projects without context loss
- Reduces anxiety about forgetting where you left off
- Creates an audit trail of decisions for future reference or team members
- Facilitates performance reviews and project retrospectives
- Supports debugging by recording what has been tried and what failed

** Daily Development Protocol
:PROPERTIES:
:CUSTOM_ID: daily-protocol
:END:
#+LaTeX: \index{daily protocol}

1. Review the Hemingway Bridge from the previous session
2. Pull latest changes from remote repository
3. Create or checkout feature branch
4. Write or update tests before implementing features
5. Implement features in small, committable increments
6. Run linter and formatter before committing
7. Write meaningful commit messages
8. Push changes and create pull request if appropriate
9. Update daily log entry
10. Write Hemingway Bridge for next session

** Debugging Protocol
:PROPERTIES:
:CUSTOM_ID: debugging-protocol
:END:
#+LaTeX: \index{debugging}

1. Reproduce the bug consistently
2. Isolate the minimal test case
3. Form a hypothesis about the cause
4. Test the hypothesis
5. Document the fix and the root cause
6. Add regression test
7. Update daily log with lessons learned
